# legacy-shield 项目总规范

> 文档版本：v1.1
> 生效日期：2026-06-18
> 适用范围：legacy-shield 所有后续版本迭代

---

## 1. 总则

本规范用于约束 legacy-shield 项目的需求、设计、开发与验收流程，确保所有改动都有据可依、有档可查、有审可验。

**核心原则**：

1. 任何代码改动前，必须先完成设计文档、执行方案与对应 Spec。
2. 所有 Spec 必须经过评审并通过后，方可进入代码开发。
3. 已归档的 Spec/设计/执行方案禁止修改；如需调整，必须通过新版本流程重新发起。
4. 开发完成后必须经过测试验收，验收通过后方可归档关闭。

---

## 2. 文档体系

### 2.1 文档层级

| 层级 | 文档 | 说明 | 状态流转 |
|---|---|---|---|
| L1 项目总规范 | `docs/specs/project-rules.md` | 本文件，定义流程与纪律 | 长期有效，修订需单独评审 |
| L2 需求文档 | `docs/specs/requirements.md`（v1.0）<br>`docs/specs/requirements-v{x}.{y}.md` | 描述产品目标、用户场景、功能需求 | 草稿 → 评审中 → 已通过 → 已归档 |
| L3 设计文档 | `docs/specs/design.md`（v1.0）<br>`docs/specs/design-v{x}.{y}.md` | 描述技术方案、模块划分、接口设计 | 草稿 → 评审中 → 已通过 → 已归档 |
| L4 执行计划 | `docs/specs/execution-plan.md`（v1.0）<br>`docs/specs/execution-plan-v{x}.{y}.md` | 拆解任务、验收标准、风险与依赖 | 草稿 → 评审中 → 已通过 → 已归档 |
| L5 阶段 Spec | `docs/specs/phases/phase-{n}-spec.md` | 按阶段/版本落地的详细实现规范 | 草稿 → 评审中 → 已通过 → 已完成，已归档 |

### 2.2 版本命名规则

- 产品版本遵循 `MAJOR.MINOR.PATCH` 语义化版本。
- `MAJOR`：架构级重构或不兼容变更。
- `MINOR`：新增功能或较大能力增强（如 v1.1 Vue 3 生态错误完整采集）。
- `PATCH`：bug 修复或文档修正，通常不需要新增 Spec，但需遵循本规范中「补丁流程」。

## 3. 改动流程

### 3.1 标准改动流程（适用于 MINOR 及以上版本）

```
需求澄清 → 需求文档 → 需求对齐会议 → 需求分解文档 → 设计文档 → 执行计划 → 阶段 Spec → Spec 评审通过 → 代码开发 → 测试验收 → 归档关闭
```

### 3.2 各阶段要求

#### 步骤 1：需求澄清

- 与用户确认改动范围、边界、优先级、非功能性要求。
- 输出《需求澄清清单》，作为需求文档输入。

#### 步骤 2：编写需求文档

- 描述背景、目标、用户场景、功能需求、验收标准。
- 明确哪些在范围内、哪些不在范围内。

#### 步骤 3：需求对齐会议

- 在编写设计文档之前，必须与用户召开正式需求对齐会议。
- 会议目标：确保双方对所有功能需求、非功能需求、验收标准、成功指标达成完全一致的理解。
- 会议输出：《需求对齐会议纪要》，需由双方签字确认（电子签名或书面确认均可）。
- 会议纪要必须作为需求文档的附录或关联文件存档。
- 详细实施指南见 [规则 1：需求对齐会议](#规则-1需求对齐会议)。

#### 步骤 4：编写需求分解文档

- 需求对齐完成后，编写《需求分解文档》。
- 将高层需求拆分为详细、可执行的需求组件。
- 包含功能分解、任务依赖关系、资源分配估算、时间线里程碑。
- 必须经项目负责人（或代理角色）评审并批准。
- 批准后方可进入设计文档编写阶段。
- 详细实施指南见 [规则 2：需求分解文档](#规则-2需求分解文档)。

#### 步骤 5：编写设计文档

- 描述技术架构、模块设计、接口变更、数据流、关键算法。
- 如涉及用户可见变更，需包含 CLI/API/UI 交互设计。

#### 步骤 6：编写执行计划

- 将设计拆解为可执行、可验收的任务。
- 明确每个任务的输入、输出、责任人（或代理角色）、验收标准、风险与依赖。

#### 步骤 7：编写阶段 Spec

- 面向开发落地的详细规范，包含：
  - 目标
  - 交付物清单
  - 实现步骤
  - 测试用例
  - 验收标准
  - 已知风险

#### 步骤 8：Spec 评审

- 必须调用 spec 评审专家团进行评审。
- 评审结论为「通过」或「不通过」。
- 不通过时必须修订后重新评审，直到通过。

#### 步骤 9：代码开发

- 仅允许在 Spec 状态为「已通过」后开始。
- 开发过程严格遵循设计文档与 Spec。
- 如需偏离 Spec，必须先升级 Spec 并重新评审。

#### 步骤 10：测试验收

- 开发完成后调用测试验收专家进行验收。
- 验收通过后方可进入归档步骤。

#### 步骤 11：归档关闭

- 将 Spec 状态改为「已完成，已归档（冻结，不再修改）」。
- 更新验收报告。

### 3.3 补丁流程（适用于 PATCH 级别修复）

对于不影响架构和接口的 bug 修复、文档修正、性能微调：

1. 仍需编写或更新对应 Spec（可在现有版本 Spec 中补充修复任务，或创建补丁 Spec）。
2. 仍需通过 Spec 评审。
3. 如涉及设计变更，需同步更新设计文档。

紧急安全修复可简化文档，但事后必须补全 Spec 与评审记录。

## 4. 规则 1：需求对齐会议

### 4.1 目的

在编写设计文档之前，通过正式会议确保项目团队（含 SOLO Coder 与用户）对需求范围、验收标准、成功指标达成完全一致的理解，避免后续设计与实现出现方向性偏差。

### 4.2 触发条件

- 《需求文档》初稿已完成。
- 下一步将进入《设计文档》编写阶段。
- 任何涉及 MINOR 及以上版本的改动均必须召开。

### 4.3 参与角色

| 角色 | 职责 |
|---|---|
| 用户 / 产品负责人 | 确认需求意图、业务优先级、验收标准 |
| SOLO Coder（主控） | 主持对齐、记录会议纪要、澄清技术可行性 |
| 需求验收专家（可选） | 校验需求完整性与可验收性 |

### 4.4 会议议程

1. **需求范围确认**：逐项回顾需求文档中的功能需求与非功能需求，确认范围边界。
2. **验收标准确认**：明确每条需求的验收标准、测试方法、通过准则。
3. **成功指标确认**：定义可量化的成功指标（如错误采集覆盖率、测试通过率、性能基线等）。
4. **风险与约束识别**：识别技术约束、依赖、时间约束、资源约束。
5. **遗留问题澄清**：对需求文档中的歧义、矛盾、遗漏进行澄清并记录。
6. **下一步确认**：确认是否进入《需求分解文档》编写阶段。

### 4.5 会议输出

必须生成《需求对齐会议纪要》，内容包括：

- 会议时间、地点（或线上会议链接）、参与人员。
- 讨论的需求条目清单及对应结论。
- 确认后的验收标准与成功指标。
- 识别的风险与应对措施。
- 遗留问题清单、责任人及解决时限。
- 双方签字确认区。

### 4.6 签字确认

- 会议纪要必须由用户方代表与 SOLO Coder 双方签字确认。
- 电子签名、邮件确认、即时通讯工具的书面确认均可接受，但需保留可追溯记录。
- 未经双方确认的会议纪要不得进入下一阶段。

### 4.7 存档要求

- 会议纪要文件命名：`docs/specs/meetings/requirements-alignment-v{x}.{y}-YYYYMMDD.md`。
- 必须在《需求文档》中引用该会议纪要，或在《需求文档》附录中附加会议纪要链接。
- 已签字的会议纪要视为项目正式资产，纳入版本控制。

## 5. 规则 2：需求分解文档

### 5.1 目的

在需求对齐完成后、设计文档编写之前，将高层需求拆分为详细、可执行、可跟踪的需求组件，为设计文档、执行计划、任务 Spec 提供清晰的输入。

### 5.2 触发条件

- 《需求对齐会议纪要》已由双方签字确认。
- 下一步将进入《设计文档》编写阶段。

### 5.3 文档内容

《需求分解文档》必须包含以下章节：

#### 5.3.1 功能分解

- 将每个高层功能需求拆分为具体的功能点。
- 每个功能点应包含：编号、名称、描述、对应高层需求编号、验收标准、优先级。
- 示例：
  - `REQ-V1.1-001` 运行时错误完整采集 → `F-001` 初始 Vue 3 app 渲染错误采集、`F-002` 动态 createApp 错误采集等。

#### 5.3.2 任务依赖关系

- 以表格或 DAG 图形式列出所有任务及其前置依赖。
- 明确哪些任务可并行、哪些必须串行。
- 示例：
  - T2（类型扩展）无依赖，可先行。
  - T1（注入脚本重构）依赖 T2。
  - T3（端到端测试）依赖 T1、T2。

#### 5.3.3 资源分配估算

- 对每个任务给出工作量估算（高 / 中 / 低 或人天）。
- 明确需要的角色（开发专家、测试专家、评审专家）。
- 明确需要的外部资源（如测试夹具、第三方库、CDN 或本地静态文件等）。

#### 5.3.4 时间线里程碑

- 列出关键里程碑及预计完成时间：
  - 需求对齐完成日期
  - 需求分解文档批准日期
  - 设计文档评审通过日期
  - 执行计划评审通过日期
  - 阶段 Spec 评审通过日期
  - 代码开发完成日期
  - 测试验收通过日期
  - 归档关闭日期

#### 5.3.5 风险与假设

- 列出需求分解过程中识别的新风险。
- 列出关键假设（如 Playwright 可用、Vue 3 全局构建版本可获取等）。

### 5.4 评审与批准

- 《需求分解文档》完成后，必须提交项目负责人（或代理角色）评审。
- 评审重点：
  - 功能分解是否覆盖全部需求。
  - 任务依赖是否合理、无循环依赖。
  - 资源估算是否可行。
  - 里程碑是否现实。
- 评审结论为「批准」或「需修订」。
- 未获批准前，不得开始《设计文档》编写。

### 5.5 存档要求

- 文件命名：`docs/specs/requirements-decomposition-v{x}.{y}.md`。
- 必须在《设计文档》中引用该分解文档。
- 已批准的《需求分解文档》纳入版本控制，状态为「已通过」。

## 6. 文档状态定义

| 状态 | 含义 | 是否可修改 |
|---|---|---|
| 草稿 | 文档初稿，尚未进入评审 | 是 |
| 评审中 | 已提交评审，等待或正在评审 | 是（仅限按评审意见修订） |
| 已通过 | 评审通过，可作为开发依据 | 否，需通过补丁流程或新版本 |
| 已完成，已归档 | 开发验收完成，冻结不再修改 | 否 |

## 9. 纪律

1. **严禁未对齐先设计**：任何 MINOR 及以上版本在进入设计文档编写前，必须先完成正式需求对齐会议并产出双方签字的会议纪要。
2. **严禁未分解先设计**：《需求分解文档》未经项目负责人批准前，不得开始《设计文档》编写。
3. **严禁未审先开发**：任何在 Spec 状态为「已通过」前开始的代码改动，均需回滚并补流程。
4. **严禁擅自修改已归档文档**：包括已归档的 Spec、设计文档、执行计划。
5. **严禁范围蔓延**：开发过程中发现的新需求，必须单独走新版本流程，不得混入当前 Spec。
6. **文档与代码一致**：开发完成后，相关用户文档（README、usage、api、custom-rules）必须与实现保持一致。

## 10. 附录：文档模板速查

### Spec 文件头模板

```markdown
# {标题}

> 版本：v{x}.{y}
> 对应需求文档：requirements-v{x}.{y}.md
> 对应设计文档：design-v{x}.{y}.md
> 对应执行计划：execution-plan-v{x}.{y}.md
> 状态：草稿
> 评审记录：见本文档末尾
```

### 状态修改示例

```markdown
> 状态：已通过
```

开发验收完成后改为：

```markdown
> 状态：已完成，已归档（冻结，不再修改）
```
