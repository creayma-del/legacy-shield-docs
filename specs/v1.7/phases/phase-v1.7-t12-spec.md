## 3. 实现步骤

### 3.1 前置条件确认

T3 已完成 pnpm workspace 配置，T2 已完成历史 spec 迁移。主仓库 `docs/specs/` 仅保留 v1.7 相关文档。

### 3.2 搜索测试文件中的 docs/ 路径引用

```bash
# 搜索 tests/ 下所有文件中对 docs/ 的引用（不限制文件类型，与设计文档 §8.2 "所有文件"一致）
grep -rn "docs/" tests/
```

记录所有匹配的文件路径、行号、引用内容。

### 3.3 分类评估

将搜索结果按类型分类：

1. **文档路径引用**：测试代码中引用 `docs/` 下的文档文件路径（如 `docs/api.md`、`docs/specs/`）
2. **vendor 文件匹配**：`tests/fixtures/` 下的第三方库 JS 文件中包含 `docs/` 字符串（非文档引用）
3. **其他匹配**：注释、字符串中的 `docs/` 提及

### 3.4 验证设计文档 §8.2 预评估结论

对照设计文档 §8.2 的预评估结论，验证：

1. 测试中是否确实无引用 `docs/` 文档路径的断言 — 验证此结论是否成立
2. `tests/fixtures/vue3/vendor/` 下的匹配是否为第三方库 JS 文件 — 验证这些文件是否为 vendor 文件
3. 若发现新的 docs/ 路径引用（设计文档预评估时可能遗漏），评估是否需调整

### 3.5 输出评估结论文档

创建 `docs/specs/v1.7/evaluations/t12-test-path-evaluation.md`，包含：

1. 评估方法与搜索结果
2. 分类评估结论（文档路径引用 / vendor 文件匹配 / 其他匹配）
3. 设计文档 §8.2 预评估结论验证结果
4. 最终结论：是否需调整测试（预期结论：不需调整，测试不引用文档路径）
5. 若发现需调整的引用，列出具体调整方案
