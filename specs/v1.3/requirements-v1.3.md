## 1. 需求背景

legacy-shield 项目定位从“针对老项目的代码质量检查”扩展为“对公司内部业务系统进行全方位监控与护航”。公司内部业务系统通常处于快速迭代、多人协作的开发阶段，往往缺少完整的 TypeScript 覆盖、全面的测试用例与统一的 ESLint 规范，因此需要在开发阶段提供完整的监控能力与日志上下文，为后续 AI 分析或人工排查提供依据。

v1.3 作为 MINOR 版本，重点在开发阶段扩展监控对象与监控维度：覆盖 Web 端与移动端 H5 业务系统，新增内存泄漏与资源加载长耗时检测能力，并将监控结果以结构化日志形式持久化。

## 2. 需求条目

| 需求编号 | 需求描述 | 优先级 | 验收要点 |
|---|---|---|---|
| REQ-1.3-1 | 监控对象覆盖 Web 端与移动端 H5 业务系统 | P0 | `quality` 子命令可识别并适配 H5/Web 项目；支持 `--platform` 显式覆盖 |
| REQ-1.3-2 | 监控重心放在开发阶段，输出完整日志为 AI 提供明确上下文 | P0 | 监控结果以 NDJSON 结构化日志持久化，字段完整、可被机器解析 |
| REQ-1.3-3 | 在 v1.2 已集成的 code-quality 能力基础上继续扩展代码静态质量检查 | P1 | 新增与内存、资源相关的静态规则，默认启用且不影响 v1.2 行为 |
| REQ-1.3-4 | 暂不监控线上运行时错误 | P0（不实现） | v1.3 不采集线上运行时 JS 错误、不建设 Sentry 类监控 |
| REQ-1.3-5 | 新增内存泄漏检测与资源加载长耗时检测 | P0 | 支持静态风险扫描 + 可选的运行时采集，输出可读报告与日志 |
| REQ-1.3-6 | 不实现依赖安全检测 | P0（不实现） | v1.3 不扫描依赖漏洞、许可证风险 |
| REQ-1.3-7 | 不内置强制公司级规范 | P0（不实现） | 保持可配置、零侵入，不强制统一 ESLint/Prettier 规范 |
| REQ-1.3-8 | 保持现有 CLI 子命令接入方式 | P0 | 在 `quality` 子命令上扩展参数，不新增 CI 插件或运行时 SDK |
| REQ-1.3-9 | 不建设在线数据看板，但需保存完整日志 | P0 | 结构化日志写入本地文件，默认保留 30 天，支持输出目录与保留策略配置 |
| REQ-1.3-10 | 所有指标不阻断发版或合并 | P0 | v1.3 新增规则默认 severity 为 warning，新增运行时监控失败不中断 `quality` 流程 |

## 3. 验收标准

1. v1.3 新增的监控能力对 Web 端与移动端 H5 项目均可运行。
2. 代码静态质量检查能力在 v1.2 基础上有所增强，且不影响原有 `quality` 子命令行为。
3. 内存泄漏检测与资源加载长耗时检测能够以 CLI 方式输出可读的报告与日志。
4. 完整日志信息可被持久化保存，便于后续 AI 分析或人工排查。
5. `pnpm typecheck`、`pnpm build`、`pnpm test` 全量通过，不引入破坏性变更。

---

## 4. 成功指标

- 开发团队能够在不修改业务系统源码的前提下，通过 legacy-shield CLI 获取完整的代码质量与性能监控报告。
- 输出日志包含足够的上下文信息，可直接用于问题定位或 AI 辅助分析。
- v1.2 的现有能力无回归。

---

## 5. 不在范围内

以下事项已在需求对齐会议中明确排除，v1.3 不予实现：

- Node.js 后端服务监控（REQ-1.3-1 备注）。
- 微信小程序、支付宝小程序等小程序端监控（REQ-1.3-1 备注）。
- 桌面端应用（Electron 等）监控（REQ-1.3-1 备注）。
- 线上运行时错误采集与告警（REQ-1.3-4）。
- 依赖安全漏洞、许可证风险检测（REQ-1.3-6）。
- 公司级强制 ESLint/Prettier/TypeScript 规范统一（REQ-1.3-7）。
- 在线数据看板、实时 Dashboard、服务端日志聚合（REQ-1.3-9）。
- 阻断发版、合并门禁或 CI 失败策略（REQ-1.3-10）。
