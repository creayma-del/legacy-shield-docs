## 1. 功能分解

|---|---|---|---|---|---|

| F-2 | 代码静态质量检查增强 | 在 v1.2 code-quality 基础上扩展规则或检测项，覆盖更多代码质量问题 | REQ-1.3-3 | 新增规则可配置、可跳过；不影响 v1.2 行为 | P1 |
| F-3 | 内存泄漏静态分析 | 通过 AST/代码扫描识别常见内存泄漏风险点（如未清理的定时器、事件监听、闭包引用等） | REQ-1.3-5 | 输出风险列表及代码位置；误报可控 | P0 |
| F-4 | 内存泄漏运行时采集 | 在 headless 浏览器或受控运行时环境中执行项目，采集实际内存指标并验证风险点 | REQ-1.3-5 | 可输出内存增长曲线与泄漏疑似点；支持 H5/Web 页面 | P0 |
| F-5 | 资源加载长耗时静态分析 | 扫描代码中可能导致资源加载耗时的写法（如大体积图片、同步脚本、不合理动态 import 等） | REQ-1.3-5 | 输出潜在资源风险及优化建议 | P1 |
| F-6 | 资源加载长耗时运行时采集 | 在浏览器环境中采集实际资源加载耗时，识别长耗时请求与阻塞资源 | REQ-1.3-5 | 输出资源加载瀑布图或结构化数据；支持 H5/Web 页面 | P0 |
| F-7 | 结构化日志输出与持久化 | 将监控结果以 JSON/NDJSON 格式写入本地文件，保留完整上下文，便于 AI 或人工分析 | REQ-1.3-2、REQ-1.3-9 | 日志字段完整、格式一致；可配置输出目录与保留策略 | P0 |

## 5. 风险与假设

| 风险 / 假设 | 说明 | 应对措施 |
|---|---|---|
| H5 与 Web 端运行环境差异 | 移动端 H5 可能存在特定 API、视口、网络环境差异，导致检测规则不适用 | 在 T1 中明确 H5 识别方式，并为 H5 场景增加独立测试夹具 |
| 运行时采集依赖 headless 浏览器 | 引入 Playwright/Puppeteer 会增加安装体积与 CI 复杂度 | 评估复用项目现有 browser/proxy 能力，或作为可选依赖 |
| 静态分析与运行时采集结果不一致 | 同一风险点可能静态提示存在但运行时未复现，或反之 | 设计文档中明确两种检测的互补关系，分别输出不同严重级别 |
| 结构化日志规模过大 | 完整上下文可能导致单项目日志文件过大 | T7 中实现日志分级、采样与按会话/项目分文件策略 |
| AI 消费方未明确 | 日志格式需兼顾可读性与机器解析 | 采用 JSON/NDJSON 标准格式，字段命名稳定并附 schema 说明 |

## 6. 已决策事项

| 事项 | 决策结论 |
|---|---|
| H5 项目识别方式 | 默认自动推断，同时支持 `--platform h5` 显式覆盖 |
| 运行时采集工具 | 先评估复用项目现有 browser/proxy 模块；能力不足时引入 Playwright 作为可选依赖 |
| 日志输出目录与保留策略 | 默认输出到 `<project>/.legacy-shield/logs/<sessionId>.ndjson`，保留最近 30 天或按配置清理 |
| 静态质量规则扩展范围 | 优先补充与内存、资源相关的规则 |
